# MCP サーバーの3種類の実装パターン

MCP（Model Context Protocol）は、AI エージェントが外部ツールやデータソースに接続するための通信規格です。
同じ MCP サーバーでも、「どこで動かすか」によって実装パターンが3種類に分かれます。

---

## パターン① ローカル PC（STDIO）

MCP サーバーが**ユーザーの手元のPCの上で動く**パターンです。

Claude Desktop などのクライアントが、PCにインストールされた `npx` や `python` コマンドを起動します。
クライアントと MCP サーバーは、標準入出力（STDIO）でやり取りします。

この構成の特徴は、MCP サーバーが PC 上で動くため、ローカルのファイルやデータベースに直接アクセスできることです。
たとえば PC 上の PostgreSQL に接続したり、ローカルのファイルを読み書きしたりすることが容易です。

一方、このパターンは「クライアントが PC 上で動いている」ことが前提です。
クラウド上の AI エージェント（IBM watsonx Orchestrate など）からは、ユーザーの PC に直接接続することはできないため、このパターンは利用できません。

---

## パターン② サーバーホスト型（StreamableHTTP）

MCP サーバーを**インターネット上の公開サーバーに置く**パターンです。

MCP サーバーを Railway、Render、AWS などのクラウドサービスにデプロイし、HTTPS で公開します。
クライアントは公開 URL を指定して HTTPS で接続します。

この構成の最大のメリットは、クラウド上の AI エージェントからも接続できることです。
watsonx Orchestrate のようなクラウドサービスから呼び出すことも可能です。

ただし、MCP サーバーをホストするためのサーバーインフラを自分で用意・運用する必要があります。
デプロイ先の選定、セキュリティ設定、ドメインや証明書の管理など、インフラの運用コストがかかります。

---

## パターン③ WxO STDIO（WxO クラウド内で動く）

MCP サーバーが**WxO のクラウド環境の中で動く**パターンです。今回採用した方式です。

IBM watsonx Orchestrate の MCP Toolkit 機能を使うと、`npx` や `python` などのコマンドを WxO のクラウド（OpenShift コンテナ）上で起動させることができます。
WxO エージェントと MCP サーバーは、クラウド内部で STDIO を使って通信します。

この構成の大きな特徴は、ユーザーが MCP サーバーを動かすための環境を一切用意しなくてよいことです。
手元の PC に Node.js をインストールする必要はありません。公開 URL を持つサーバーを用意する必要もありません。
MCP サーバーのコマンド（例: `npx @modelcontextprotocol/server-postgres`）を Toolkit の設定ファイルに書くだけで、WxO が自動的に起動・管理します。

データベースへの接続情報（DATABASE_URL など）は WxO の Connection 機能でセキュアに管理され、MCP サーバープロセスの環境変数として自動的に注入されます。

---

## 3つのパターンの比較

| | ① ローカル PC | ② サーバーホスト型 | ③ WxO STDIO |
|---|---|---|---|
| MCP サーバーの実行場所 | ユーザーの PC | 自前のクラウドサーバー | WxO クラウド内 |
| クラウドエージェントから利用 | ❌ 不可 | ✅ 可能 | ✅ 可能 |
| PC への Node.js 等のインストール | 必要 | 不要 | 不要 |
| 公開サーバーの用意・運用 | 不要 | 必要 | 不要 |
| ローカルファイルへのアクセス | ✅ 容易 | ❌ 困難 | ❌ 困難 |
| 設定の手軽さ | アプリに依存 | インフラ知識が必要 | YAML 設定のみ |

---

## まとめ

ローカル PC パターンは Claude Desktop などのローカルアプリと組み合わせるときに便利ですが、クラウドエージェントからは使えません。
サーバーホスト型はどこからでも使えますが、サーバーの運用が必要です。
WxO STDIO パターンは、PC も公開サーバーも不要で、WxO エージェントと MCP を最も手軽に組み合わせられる方法です。
